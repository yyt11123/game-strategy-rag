# 🎮 游戏攻略智能问答助手 — 原理说明

> 本文档用**大白话**讲解本项目涉及的核心技术原理，可以直接作为 PPT 讲稿素材使用。

---

## 一、大模型为什么会"一本正经地胡说"？（幻觉问题）

### 什么是"幻觉"（Hallucination）？

如果你直接问 ChatGPT 或通义千问一个它训练数据里没有的问题，比如：

> "《幻境传说》里的暗影领主用什么阵容打？"

它可能会编一个听起来很合理、但完全错误的答案——这就是**幻觉**。

### 为什么会幻觉？

大语言模型（LLM）本质上是一个**超级复杂的"下一字预测"机器**：

1. 训练时，它读了几万亿字的文本，学会了"在某个上下文后面，下个字最可能是什么"。
2. 但它**并没有记住**所有训练数据——它学的是语言规律和模式，而不是一个精确的数据库。
3. 当被问到一个它不知道答案的问题时，它不会说"我不知道"（因为它没有被教这样做），而是会基于语言规律**生成一个"看起来像正确答案"的回答**。

打个比方：大模型就像一个博览群书但记忆力模糊的人——你问他一个冷门知识，他可能凭印象瞎编一个听起来很合理的答案，而不是承认自己不记得。

### 那怎么解决？

两种主流方案：
- **微调（Fine-tuning）**：用特定领域数据继续训练模型——费钱费时，学生作业不现实。
- **RAG（检索增强生成）**：不给模型"灌知识"，而是在提问时把相关资料"塞"给它——本项目采用的方式。

## 二、RAG 如何用"先检索资料再回答"缓解幻觉？

### RAG 的核心思想

**不要让模型凭空回忆答案，而是先把答案所在的资料找出来，和问题一起交给模型。**

类比：开卷考试 vs 闭卷考试。
- 闭卷考试（直接问大模型）→ 可能瞎编
- 开卷考试（RAG）→ 把课本翻到相关页，照着答

### RAG 的五个步骤

```
攻略文档(txt/pdf)
   ↓ ① 读取（Document Loader）
   ↓ ② 切块（Splitter，把长文档切成小段）
   ↓ ③ 向量化（Embedding，把每段文字转成一串数字）
   ↓ ④ 存入向量库（Vector Store）
─────────────── 以上是"建库"，启动时做一次 ───────────────
用户提问
   ↓ ⑤ 把问题也向量化，在向量库里检索最相关的片段
   ↓ ⑥ 把【检索到的片段 + 用户问题】一起交给大模型
   ↓ ⑦ 大模型基于提供的资料生成回答（附带来源）
```

**关键：** 大模型不需要"知道"攻略内容——攻略内容在每次提问时作为参考资料临时喂给它。这样即使模型训练数据里完全没有这个游戏，它也能正确回答。

### 对应到本项目代码

| 步骤 | 代码位置 | 关键方法 |
|------|----------|----------|
| ① 读取 | `KnowledgeBase.java` → `loadAllDocuments()` | 读 txt（`Files.readString`）/ PDF（`ApachePdfBoxDocumentParser`） |
| ② 切块 | `KnowledgeBase.java` → `build()` | `DocumentSplitters.recursive(500, 100)` 每500字一块，重叠100字 |
| ③ 向量化 | `KnowledgeBase.java` → `build()` | `embeddingModel.embed(segment)` 调用 text-embedding-v4 |
| ④ 入库 | `KnowledgeBase.java` → `build()` | `embeddingStore.add(embedding, segment)` 存入内存向量库 |
| ⑤ 检索 | `KnowledgeBase.java` → `search()` | `embeddingStore.search(...)` 按语义相似度查找 |
| ⑥ 生成 | `RagService.java` → `ask()` | 拼 Prompt（系统指令+参考资料+问题）→ 发给 qwen-plus |
| ⑦ 来源 | `RagAnswer.java` → `toFormattedString()` | 回答末尾附上文件名 + 命中片段 |

## 三、什么是 Embedding（向量化）？

### 为什么要把文字变成向量？

计算机不懂中文，它只懂数字。要让计算机判断两段文字"语义是否相近"，必须先把文字翻译成它能比较的形式——**向量（一长串数字）**。

### 向量是什么？

向量就是一组数字，比如一个 1024 维的向量：
```
[0.023, -0.451, 0.789, ..., 0.134]   ← 共 1024 个数字
```

这段数字可以理解为文字在**高维语义空间中的坐标**。

### 向量怎么能"按语义找相似内容"？

**核心原理：语义相近的文字，向量也相近。**

比如在一个训练好的向量空间中：
- "打 Boss 用什么阵容"的向量 ≈ "推荐阵容"的向量（语义相近，距离近）
- "打 Boss 用什么阵容"的向量 ≠ "今天天气真好"的向量（语义不同，距离远）

两个向量的相似度通常用**余弦相似度（Cosine Similarity）**计算，值在 -1 到 1 之间，越接近 1 越相似。

### 一个直观的类比

把向量想象成地图上的经纬度坐标：
- "北京"的坐标是 (39.9°N, 116.4°E)
- "天津"的坐标是 (39.1°N, 117.2°E)
- 这两个坐标很近 → 北京和天津很近

向量化模型（如 text-embedding-v4）就是这个"翻译官"：输入一段文字，输出它在语义地图上的坐标。

### 本项目用的是远程 Embedding API

`text-embedding-v4` 是阿里云百炼提供的远程向量化服务。每次调用都会：
1. 把文字发送到阿里云服务器
2. 服务器用训练好的模型计算向量
3. 返回 1024 个数字的向量结果

**所以建库（把所有攻略块向量化）和每次提问都需要联网、消耗额度。**

## 四、Agent 与 RAG 的区别

### 一句话总结

| | RAG（检索增强生成） | Agent（智能体） |
|---|---|---|
| **决策者** | 程序员（写死了流程） | 大模型（自主决定） |
| **流程** | 每次提问必先检索 | 模型先分析，再决定调哪个工具 |
| **工具** | 只有一个"查攻略" | 多个工具可选（查攻略、计算、搜索等） |
| **类比** | 流水线工人——按固定步骤操作 | 项目经理——根据任务自己调度资源 |

### 普通 RAG 流程（被动、固定）

```
用户提问
  → [必须] 检索知识库
  → [必须] 把检索结果 + 问题给大模型
  → 生成回答
```

不管用户问什么（哪怕问"1+1等于几"），RAG 都会先去查攻略——这显然是浪费。

### Agent 流程（主动、自主）

```
用户提问
  → 大模型分析问题：
      "这是一个攻略问题"  → 调用"查攻略工具" → 拿到攻略内容
      "这是一个计算问题"  → 调用"计算工具"   → 拿到计算结果
      "既要查又要算"      → 先调查攻略，再调计算
  → 综合所有工具结果，生成最终回答
```

### 本项目 Agent 怎么决定调哪个工具？

实现基于 OpenAI 的 **Function Calling（函数调用）** 机制，通义千问完全支持：

1. **程序告诉大模型**："你有两个工具可用——查攻略（输入：问题描述）、计算器（输入：运算类型和数值）"
2. **大模型分析用户问题**，判断需要哪个/哪些工具。
3. 如果需要，大模型返回一个**函数调用请求**（不是文字回答），内容是"请调用工具 X，参数是 Y"。
4. **程序执行**对应的 Java 方法，把结果返回给大模型。
5. 大模型基于工具结果**生成最终的文字回答**。
6. 如果还需要更多信息，重复步骤 3-5（循环直到大模型满意）。

### 对应到本项目代码

| 组件 | 代码位置 |
|------|----------|
| 查攻略工具 | `agent/tools/StrategyTool.java` — `@Tool` 注解的 `searchStrategy()` 方法 |
| 计算工具 | `agent/tools/CalculatorTool.java` — `@Tool` 注解的 `calculate()` 方法 |
| Agent 组装 | `agent/GameAssistantAgent.java` — `AiServices.builder()` 注册工具并创建 Agent |

### 为什么 Agent 更适合复杂场景？

因为真实用户的问题往往不是单一的：
- "我需要做 3 件暗系装备，每件要 10 个暗影精华，暗影领主一次掉 3~5 个，我得刷几次？"

这个问题需要：
1. 查攻略 → 确认暗影精华的掉落数量（3~5 个/次）
2. 计算 → 30÷4≈8 次

**RAG 做不了**（它只会查攻略，不会算）。**Agent 可以先查后算**，给出完整的结论。

## 五、常见答辩问题（Q&A 素材）

### Q1：为什么不用关键词搜索，而要用向量搜索？

关键词搜索只能匹配"字面相同"的内容。用户问"打 Boss 用什么阵容"，攻略里写的是"推荐阵容"——关键词搜不到，但向量搜索能根据语义找到。

### Q2：向量化（Embedding）消耗额度吗？

消耗。每次调用 `text-embedding-v4` API 都会按 token 数计费。建库时每块调用一次（文档越长、切块越多、消耗越大），每次提问检索时再调用一次。

### Q3：如果攻略内容很多（几百个 PDF），怎么办？

当前内存向量库适合小规模（几千条以内）。大规模应用需要：
- 换用持久化向量数据库（如 Chroma、Milvus、Weaviate）
- 但原理完全一样，只是换了存储后端

### Q4：Agent 会不会选错工具？

有可能。如果问题模糊，模型可能判断失误。优化方法：
- 把工具的描述（`@Tool` 的 value 和参数的 `@P` 注解）写得更精确
- 使用更强的模型（如 qwen-max 代替 qwen-plus）
- 在 System Prompt 里给出更多判断规则

### Q5：为什么不用本地 Embedding 模型省钱？

本地模型（如通过 ONNX 运行）不需要联网也不消耗额度，但：
- 对电脑配置有要求（需要一定内存和 CPU）
- 配置复杂，不适合"开箱即跑"的作业演示
- API 调用的效果通常好于本地小模型

### Q6：大模型怎么保证只根据攻略回答，不编造？

通过 System Prompt 约束：
> "只能根据下面提供的参考资料来回答问题。如果参考资料中找不到相关信息，请明确回复'攻略中未找到相关信息'，绝对不要编造。"

但这只是"软约束"，无法 100% 保证。项目还通过"相关性阈值"（`RELEVANCE_THRESHOLD`）过滤不相关的检索结果，进一步减少幻觉空间。
